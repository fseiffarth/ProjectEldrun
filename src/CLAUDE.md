| `embed/YamlTree.tsx` | The YAML/JSON structure tree: the editable half of the viewer, the way the rendered preview is markdown's. It renders *rows*, but it edits *text* — every action splices the draft `TextView` already owns, so Tree and Source need no conversion between them and a tree edit is an ordinary dirty/undoable/saveable change. A flow (JSON) container is badged as such, because that is the style its edits will keep. A key wears its `#` comment on its **hover** and a `#` marker when it has one (a config file's comments are the half worth reading, so they are not hidden behind a hover — only their text is). Reordering is a **pointer** drag from the row's grip (HTML5 DnD does not work under WebKitGTK, and mid-gesture listeners get dropped, so every handler is bound to the grip up front); it lives in `YamlRows` because a node can only be reordered **among its siblings** — dragging a key into another mapping is a re-parent, not a reorder. Every entry also has a **copy** button feeding an app-wide buffer (plus the system clipboard as text) and a click-placed **paste cursor**: the cursor only lands where the entry *fits* (`canPasteAfter` — mapping entry into a map, item into a list, block/flow never mixed), so a paste can never tear the file. |
| `embed/YamlGrid.tsx` | The YAML/JSON **card view** — the **default** view for structured YAML/JSON, beside Tree and Source (#yaml-grid; the toggle reads "Cards"). A **drill navigation, not an infinite nest**: one card is "main" at a time; the grid shows a breadcrumb from the document down to it, the main card's *level* (its siblings, so you can step sideways without going up first), and the main card's *children* as the next level below. A card shows only its own **scalar fields** — its nested collections are reached by clicking its title (or its "N groups" footer), which opens it as the new main and reveals them as the next level; a card with no nested collection gets a static (non-clickable) title. With nothing focused the grid is the top overview: a root map is one `document` card, a root list is a card per item. A list item's card is titled by its name-ish field (`name`/`id`/…) or `Item N`, never `#0`. **Cards reorder by dragging** among their siblings — pointer-based from a grip (HTML5 DnD is dropped by WebKitGTK) — committed as one `moveNodeTo` splice, exactly as `YamlRows` does. Like `YamlTree` it renders controls but edits *text* — every action splices the same draft via `lib/viewers/yaml`'s ops (`setValue`; `addChild` for `+ field`/`+ card`/`+ value`; `deleteNode`; `duplicateNode`; `moveNodeTo`), so a card edit is an ordinary dirty/undoable/saveable change and Cards/Tree/Source are three views on one text. A value it can't safely rewrite is shown, not offered (`source only`). `TextView` opens YAML in Cards by default and falls back to Tree for a flat file (no `hasCards`), where the Cards toggle is hidden. |
| `embed/TableView.tsx` | The CSV/TSV table (and, read-only, the `.xlsx` sheet). Like `YamlTree`, it renders *rows* but edits *text* — a cell edit splices the draft `useEditableFile` owns, so it is an ordinary dirty/undoable/saveable change and every cell nobody touched keeps its bytes. Hence sort and filter carry each row's **source** index (`RowRef`): a splice must address the row a cell came *from*, not the row it sits on. The separator is a **guess the reader can overrule** — sniffed, then offered in the header (Auto/comma/semicolon/tab/pipe/custom); an override persists per tab, Auto does not. Columns hide by name from `ColumnsMenu` (click to hide, click again to show; multi-select, so it stays open), which is why the row filter searches only the **visible** columns and why hiding is dropped when the delimiter changes — a hidden column is an *index*, and another separator cuts the row into different ones. Only the visible rows render, which is why the column widths are measured over the whole file up front: sized to what is on screen, every column would resize as you scrolled. |
| `embed/GifView.tsx` | The animated-GIF viewer (#gifviewer): the plain image viewer already *animates* a GIF (the webview animates `<img>` natively) but offers no control, so this one decodes the frames itself (`lib/viewers/gif.ts`) onto a `<canvas>` and adds the transport — play/pause, frame stepping, scrubber, speed, loop, frame/delay readout — over the image viewer's exact zoom/pan (shared exported pieces: `zoomOffset`/`clampScale`/`useNonPassiveWheel`). Playback is a rAF timestamp accumulator, so per-frame delays are honored without drift; authored sub-20ms delays play at the 100ms browser convention. A GIF the decoder can't handle degrades to the native animated `<img>` — never worse than before the viewer existed. No Annotate: flattening to PNG would silently destroy the animation. |
| `embed/pdf/PdfViewer.tsx` | The PDF viewer: pdf.js canvas stack, zoom/find/print toolbar, SyncTeX, page rail, and the **contents** sidebar (chapters, ☰ at the toolbar's far left). Renders an *arrangement* (`pageModel`), not the file — so reorder/delete/turn/merge are array ops, and nothing is written until Save. |
| `embed/pdf/pdfDoc.ts` | The open source documents an arrangement draws from, and `buildPdf` — the ONLY place a PDF is written (pdf-lib, on save). |
| `embed/pdf/outline.ts` | The contents/chapters model (#pdf-outline), pure. `loadOutline` resolves the PDF's **embedded** outline (bookmarks) to 1-based file pages (named + explicit dests); `detectHeadings` is the **font-size fallback** used only when there's no embedded outline — it infers a heading tree from the text's typography (body-size mode, line grouping, running-header drop, nest-by-size). Jumps map a file page to whichever sheet shows it, so they survive page rearrangement. Tested in `src/__tests__/PdfOutline.test.ts`. |
| `common/PageStrip.tsx` | **The** page-thumbnail strip: drag-reorder, shift-select, turn, delete, right-click. Used twice — horizontally by the print preview, vertically as the PDF viewer's page rail. |
| `common/mountPageStrip.tsx` | `createRoot` adapter letting the imperative print modal host `PageStrip`. |

Passes the file's **absolute** path as the script id so one command works remote or local. Untested until cluster QA. |
| `lib/pythonRun.ts` | Run/Debug a Python file (#87) by opening a **terminal tab** — not a bespoke exec path. That is what makes it inherit remote-host and container locality for free (the *tab* carries both) and stay a real interactive terminal (`input()`, `Ctrl+C`, output survives exit). Debug is pdb pre-loaded with the gutter's breakpoints (`-c "b f:N" … -c continue`). **Which** Python it runs is asked of the backend (`python_interpreter_for`), never re-derived here: two interpreter rankings that can disagree is a bug waiting to happen, and only the backend can see conda/poetry at all. |
| `lib/experimental.ts` | The experimental-flag gate: **off for everyone, on in debug mode** (unset ≠ false — it falls back to `settings.debug`, so a flag still moving is invisible to someone *using* Eldrun and present by default for someone *building* it). Read a flag via `useExperimental`, never `settings.<flag> ?? false` — that spelling is what makes a flag miss the debug default. Gates `agent_mode_toggle` and `python_run_debug`. |
| `lib/tmuxSession.ts` | Persistent remote sessions (TODO #85), the frontend mint + decision helpers. `newTmuxSessionName()` mints the stable `eldrun-<uuid>` a shell tab persists as `TabEntry.tmuxSession` (minted once at creation — the tab's PTY id is *not* stable, `loadFromLayout` regenerates the key, so a derived name would fork a second session on relaunch instead of reattaching). `persistSessionsEnabled` encodes **default ON** (`persist_sessions !== false`); `shouldPersistTab` = a **shell** tab (Python runs open one) on a **remote host** of a persist-enabled project; `shouldPersistLocalTab` = a shell tab running **locally** (a local project, or a remote project's mirror tab) with `persist_local_sessions` on (Unix) — for surviving an Eldrun crash. Never agents, never root. The actual tmux wrap is the backend's; this only mints the name + picks *which* tabs get it. |
| `lib/closeRemoteTab.ts` | The tab-close (detach) path for a persistent tab (TODO #85). Closing a tab is **non-destructive**: `closeTabWithConfirm` just `removeTab`s, which unmounts the pane and kills only the ssh/PTY *client*, so the tmux session lives on under its daemon and stays reattachable from the Sessions view — a quit/disconnect/crash/respawn all leave it alive too. The *only* way to terminate a session is its **×** (kill) in the Sessions view (`remote_tmux_kill`/`local_tmux_kill`). `persistentSessionOf`/`localPersistentSessionOf` classify which session a tab owns (the Sessions view marks owned rows). Reads the tabs/projects/settings stores, so it lives here — `stores/projects` already imports `stores/tabs`, so a store-level import back would be a cycle. |
| `lib/viewers/yaml.ts` | The YAML/JSON tree model + its edit ops, all pure. The tree is a **view on the text**: every edit is a surgical splice, never a re-serialization of the parsed model — the only way a config file's **comments** survive an edit, and what lets Tree and Source share one draft. **Both** YAML syntaxes are first-class: *block* (indentation) and *flow* (`{a: 1}` — i.e. **JSON**, inline or spread over lines). Which one a node is written in decides how it is edited (block → line splice, flow → span splice) and the tree keeps the author's choice, so `[a, b]` grows to `[a, b, c]` and is never quietly rewritten into block. `strict` (a `.json` file) is the only dialect knob: no plain scalars, so keys and strings are quoted. **Comments are a node's own, not scenery** — a `#` is the only place YAML documents a key, so each node reads the one behind its value *and* the run directly above it at its own indent, and `setComment` writes back wherever the author already wrote (a new one goes behind, which adds no line). The consequence is in the edits: a comment **travels with its node**, so deleting or reordering a key takes its description along rather than leaving it to re-attach itself to whichever key slid into that place. `moveNodeTo` is the one reorder op (the ↑/↓ buttons are `to = at ± 1`): a drag across five rows is **one** splice, not five swaps — every splice invalidates the offsets after it. A construct it could not rewrite without botching it (anchor, alias, merge key, plain scalar continued across lines) still *renders* but comes back `editable: false`; a line it can't classify at all fails the parse and defers to Source. |
| `lib/viewers/yamlGrid.ts` | The YAML/JSON **card grid** helpers (#yaml-grid), all pure. `hasCards` is the toggle gate (true when a collection nests another); `isContainer`/`cellKind`/`nestedLabel` classify a node so the card view never offers a control it can't honour — every EDIT is still delegated back to `yaml.ts`'s splice ops (the view-on-text bargain: comments/quoting/untouched bytes survive). Also keeps the **tabular** model (`gridModelFor`/`gridCandidates`/`hasGrid`): the flat rows × columns view of a single list/map-of-records — the card view doesn't render it, but it stays a tested, reusable "is this region a table?" helper. Tested in `src/__tests__/YamlGrid.test.ts`. |
| `lib/viewers/table.ts` | The CSV/TSV model + its edit ops, all pure. Two things it exists to say. **The separator is guessed, not known** — `.csv` never states which character separates the columns, so assuming comma from the extension is exactly what renders a `;`-delimited export as one tall column; `sniffDelimiter` scores each candidate by parsing a sample *with* it (a comma inside a quoted field cannot fool the `;` reading) and asking how rectangular the result is, rejecting any character that never splits a row however consistently it fails to appear. And **the table is a view on the text**: `parseTable` hands back each cell's source span, so `replaceCell`/`insertRowAfter`/`deleteRow` are surgical splices — the only way editing one cell leaves every other field's quoting, the file's line endings and its delimiter byte-for-byte alone. Same bargain `yaml.ts` strikes for comments. |
| `lib/viewers/gif.ts` | The GIF decoder, all pure — takes a `Uint8Array`, never touches the fs (WebKitGTK has no `ImageDecoder` API, so frame control means decoding LZW + interlace + disposal 0–3 ourselves; GIF89a is frozen, so no maintenance tail). Two ideas: **every frame composites to a full-canvas RGBA buffer** (GIF has no keyframes — frame N is a patch on N−1's state, so upfront full frames are what make scrubbing O(1); memory is bounded by `maxPixelBytes`, beyond which the result says `truncated`), and **delays are stored as authored, played by convention** (`effectiveDelayMs`: <20ms means "browser default" = 100ms — files were authored against that clamp). Disposal 2 clears to *transparent*, not the spec's background color, because that is what every browser does. Corruption after ≥1 good frame returns the clean prefix + `truncated`; before it, throws. |

| `lib/gpu.ts` | The GPU-memory arithmetic shared by its three surfaces (header row, system monitor, local-model menu), pure; sampling is the backend's `gpustat`. The one idea: **GPU memory is the two pools summed** — the dedicated VRAM carve-out *and* the shared pool mapped from system RAM — because on an APU the carve-out is only the framebuffer (~86% full forever, meaningless) while a model actually lands in the shared half; on a discrete card the shared half is 0 and the sum is plain VRAM. Hence `gpuTone` tones by **ratio**: the old row's absolute 1 GB/2 GB thresholds suited a figure counting only Ollama's models and would peg a device-wide one at "hot" permanently. Also home to the **sensor** formatters (`formatTempC`/`formatWatts`/`formatMhz`/`formatFan`/`gpuLinkLabel`) and the `GpuProc` type: each formatter returns `null` for an unreported reading so the UI omits it rather than print a fake zero, and `gpuAdapterTooltip` appends only the sensors the driver actually gave. The system-monitor pane renders these as a per-adapter GPU card (VRAM + Util meters, sensor strip) plus a once-only per-process GPU-memory list (locally from `gpu_process_snapshot`; for a remote source, from the host-sampled `SystemSnapshot.gpu_procs` in the same poll). |
